01 // Engineering Objective

Primary Engineering Question

How do we design a data model to calculate OEE with a 15-minute SLA for end-of-shift reporting, using our existing on-premise infrastructure?

10
Min Update Cycle
OS Task Scheduler runs Python extraction scripts every 10 minutes, feeding a T-SQL aggregation that completes well within the 15-minute SLA. Pre-aggregated summary tables ensure Power BI loads in near-instant time without hammering raw PLC cycle rows.

Ignition SCADA

Direct machine fault logging. Captures downtime events with timestamps and reason codes. Handles micro-stops under 5 minutes — events operators historically missed on paper clipboards.

SAP ERP

Source of truth for shift schedules (planned minutes), quality outcomes (good units, scrap counts), and production orders. Queried via RFC/API every 10 minutes.

MS SQL Server

Central on-premise data warehouse. Hosts the star schema, staging tables, and stored procedures. SQL Server Agent drives the Transform step on a tight schedule.

02 // The OEE Framework
OEE = Availability × Performance × Quality
Availability
run_mins ÷ planned_mins

Measures planned production time against actual operating time. Tracks major downtime events, setups, and scheduled changeovers. Ignition SCADA provides the downtime log; SAP provides the planned shift window.

Sources: Ignition SCADA Logs + SAP Shift Schedules

Performance
actual_rate ÷ ideal_rate

Evaluates machine speed vs. standard cycle rate. Accounts for speed losses, idling, and micro-stops — events under 5 minutes that operators routinely failed to manually log, now captured automatically by SCADA/PLC.

Sources: PLC Sensor (direct to MS SQL) + Ignition SCADA

Quality
units_good ÷ units_total

Measures first-pass yield logic. Captures manufactured items that do not meet quality standards, relying heavily on defect and scrap counts extracted from SAP Quality Module each cycle.

Source: SAP Quality Module (QM)

05 // Data Pipeline Workflow
01
Extract

OS Windows Task Scheduler fires Python scripts every 10 minutes. Scripts call SAP RFC/BAPI endpoints and the Ignition SCADA SDK to pull downtime logs, quality data, and shift schedules.

python 3.9 · pyodbc · requests · pyrfc
02
Stage

Python securely loads raw JSON/CSV payloads directly into staging tables in MS SQL Server — stg_scada_downtime, stg_sap_quality. No transformation here.

ms sql server 2019 · bulk insert · pyodbc
03
Transform

SQL Server Agent triggers T-SQL Stored Procedures. Cleans timestamps, resolves dim keys, merges into fact_production_event, then calls pre-aggregation SP to refresh summary table.

t-sql stored procs · sql server agent jobs
04
Report

Power BI Desktop connects via On-Premises Data Gateway to oee_shift_summary. DirectQuery or scheduled import — near-instant load from pre-aggregated data.

power bi · on-premises gateway · directquery

Ignition SCADA — Downtime Feed

Ignition logs machine faults directly to its internal SQLite/MySQL historian. The Python extractor connects via Ignition's Gateway REST API or JDBC bridge and pulls the downtime event log on a 10-minute cycle.

  • Event timestamp (start/end) → down_mins computed
  • Reason code mapped to dim_reason.reason_id
  • Machine tag maps to dim_machine.machine_id
  • Micro-stops (<5 min) surfaced automatically — no operator input
  • Planned vs. unplanned flag set by dim_reason.is_planned

SAP ERP — Quality & Schedule Feed

Python calls SAP via pyrfc (RFC-based) or the SAP OData REST API. Two datasets are pulled per cycle:

  • QM Module: inspection lot results, scrap counts, good units by production order
  • PP / HR: active shift schedules, planned production minutes
  • Material number → dim_product.sap_material lookup
  • Work center ID → dim_machine.sap_work_center lookup
  • Ideal cycle rates loaded once; refreshed on engineering change order
06 // ETL Incorporation Options

The 2021 on-prem stack uses a lightweight Python + SQL Server Agent approach. If ETL requirements grow — more sources, data quality gates, lineage, replay capability — the following options can be introduced without replacing the star schema.

Option A — Current: Python + SQL Server Agent
Recommended for 2021 Zero new licenses On-premise only

Scheduled Python scripts pull from SCADA/SAP, write to staging tables, SQL Server Agent fires stored procs. Fits inside the firewall. Lowest complexity, lowest ops overhead. Right-sized for this use case.

Best for: ≤5 data sources, stable schema, ops team comfortable with SQL

Option B — SSIS (SQL Server Integration Services)
Native MS Stack Visual pipeline designer Already licensed

Replace Python scripts with SSIS packages managed in SQL Server Data Tools (SSDT). Adds GUI-based error handling, data flow logging, and SSIS catalog for deployment. Can call SAP OData connectors via ADO.NET.

Best for: team already using SSMS, need visual debugging, more transform logic

Option C — Apache Airflow (Self-Hosted)
Open source DAG-based orchestration Python-native

Self-hosted Airflow replaces Task Scheduler. Provides DAG lineage, retry logic, SLA alerting, and a web UI for monitoring runs. Python operators reuse existing extraction code. Runs on a dedicated on-prem VM.

Best for: scaling to 10+ sources, need SLA alerting, ops team is Python-first

Option D — dbt Core + Python (Future State)
Post-2021 upgrade path SQL-first transforms Data lineage DAG

Replace stored procedures with dbt models targeting MS SQL Server. Staged → fact layer transformations become version-controlled SQL with auto-documentation and column-level lineage. Python + Airflow still handles extraction.

Best for: data engineering team, CI/CD culture, schema drift management needed

07 // T-SQL Reference
Stored Procedure — OEE Shift Summary RefreshT-SQL
CREATE OR ALTER PROCEDURE dbo.usp_RefreshOEESummary
AS
BEGIN
    SET NOCOUNT ON;

    -- Truncate & re-populate summary table (10-min SLA compatible)
    TRUNCATE TABLE dbo.oee_shift_summary;

    INSERT INTO dbo.oee_shift_summary (
        machine_id, shift_id, product_id,
        planned_mins, run_mins, down_mins,
        units_total, units_good,
        actual_rate_avg, ideal_rate,
        availability, performance, quality, oee
    )
    SELECT
        f.machine_id,
        f.shift_id,
        f.product_id,
        s.planned_mins,
        SUM(f.run_mins)                                          AS run_mins,
        SUM(f.down_mins)                                         AS down_mins,
        SUM(f.units_total)                                       AS units_total,
        SUM(f.units_good)                                        AS units_good,
        AVG(f.actual_rate)                                       AS actual_rate_avg,
        m.ideal_cycle_rate,

        -- Availability: run_mins / planned_mins
        CAST(SUM(f.run_mins) * 1.0 / NULLIF(s.planned_mins, 0) AS DECIMAL(5,4))  AS availability,

        -- Performance: avg actual_rate / ideal_cycle_rate
        CAST(AVG(f.actual_rate) / NULLIF(m.ideal_cycle_rate, 0) AS DECIMAL(5,4))  AS performance,

        -- Quality: good units / total units
        CAST(SUM(f.units_good) * 1.0 / NULLIF(SUM(f.units_total), 0) AS DECIMAL(5,4))  AS quality,

        -- OEE = A × P × Q
        CAST(
            (SUM(f.run_mins) * 1.0 / NULLIF(s.planned_mins, 0))
            * (AVG(f.actual_rate) / NULLIF(m.ideal_cycle_rate, 0))
            * (SUM(f.units_good) * 1.0 / NULLIF(SUM(f.units_total), 0))
        AS DECIMAL(5,4))                                                         AS oee

    FROM       dbo.fact_production_event  f
    INNER JOIN dbo.dim_shift              s  ON  s.shift_id        = f.shift_id
    INNER JOIN dbo.dim_machine            m  ON  m.machine_id      = f.machine_id
    WHERE  f.event_ts >= DATEADD(DAY, -30, GETDATE())  -- rolling 30-day window
    GROUP BY   f.machine_id, f.shift_id, f.product_id, s.planned_mins, m.ideal_cycle_rate;

END;
Python Extract — Ignition SCADA DowntimePython 3.9
import requests, pyodbc, pandas as pd
from datetime import datetime, timedelta

# ── Config ─────────────────────────────────────────────────
IGNITION_BASE  = "http://ignition-server:8088/data/historian"
SQL_CONN       = "Driver={SQL Server};Server=MSSQL-PROD;Database=OEE_DW;Trusted_Connection=yes;"
LOOKBACK_MINS  = 15

def extract_scada_downtime():
    since = (datetime.now() - timedelta(minutes=LOOKBACK_MINS)).isoformat()

    # Pull downtime events from Ignition REST API
    resp = requests.get(
        f"{IGNITION_BASE}/alarms/events",
        params={"startDate": since, "state": "cleared", "priority": "high"},
        timeout=10
    )
    events = resp.json()["data"]

    df = pd.DataFrame(events)[[
        "eventId", "sourceTag", "ackTime", "clearTime", "reasonCode"
    ]]
    df["down_mins"] = (
        pd.to_datetime(df["clearTime"]) - pd.to_datetime(df["ackTime"])
    ).dt.total_seconds() / 60
    df["load_ts"] = datetime.now()

    # Stage into SQL Server
    conn = pyodbc.connect(SQL_CONN)
    cursor = conn.cursor()
    for _, row in df.iterrows():
        cursor.execute("""
            INSERT INTO dbo.stg_scada_downtime
              (event_id, source_tag, ack_time, clear_time, reason_code, down_mins, load_ts)
            VALUES (?,?,?,?,?,?,?)
        """, row["eventId"], row["sourceTag"], row["ackTime"],
               row["clearTime"], row["reasonCode"], row["down_mins"], row["load_ts"])
    conn.commit()
    conn.close()

if __name__ == "__main__":
    extract_scada_downtime()
Python Extract — SAP QM Quality Data via pyrfcPython 3.9
import pyrfc, pyodbc
from datetime import datetime, timedelta

SAP_CONN = {"ashost": "sap-prod.plant.local", "sysnr": "00",
            "client": "100", "user": "svc_oee", "passwd": "***"}

def extract_sap_quality():
    conn    = pyrfc.Connection(**SAP_CONN)
    results = conn.call(
        "BAPI_INSPOPER_GETLIST",
        PLANT      = "P001",
        DATE_FROM  = (datetime.now() - timedelta(minutes=15)).strftime("%Y%m%d"),
        DATE_TO    = datetime.now().strftime("%Y%m%d")
    )
    conn.close()

    sql_conn = pyodbc.connect(SQL_CONN)
    cursor   = sql_conn.cursor()
    for lot in results["INSPOPERLIST"]:
        cursor.execute("""
            INSERT INTO dbo.stg_sap_quality
              (insp_lot, material, work_center, total_qty, ok_qty, scrap_qty, load_ts)
            VALUES (?,?,?,?,?,?,?)
        """, lot["INSPOPER"], lot["MATNR"], lot["ARBPL"],
               lot["SOLLMENGE"], lot["ISTMENGE"], lot["AUSSCHUSS"],
               datetime.now())
    sql_conn.commit()
    sql_conn.close()
08 // Power BI Dashboard Design

Architecture

  • On-Premises Data Gateway installed on SQL Server host
  • Power BI Desktop connects via Gateway to oee_shift_summary
  • Scheduled refresh every 10 minutes — matches extraction cycle
  • No DirectQuery on raw fact table (millions of PLC rows) — summary only
  • Row-level security by production line / department

Recommended Visuals (End-of-Shift Report)

  • KPI cards: OEE %, Availability %, Performance %, Quality %
  • Clustered bar: OEE by machine line, current shift vs. target
  • Waterfall chart: OEE loss decomposition (A, P, Q losses)
  • Pareto: downtime by reason code — planned vs. unplanned
  • Trend line: 30-day rolling OEE per machine
  • Table: micro-stop frequency by machine + shift
Power BI DAX — OEE MeasuresDAX
-- Availability measure
Availability % =
DIVIDE(
    SUM( oee_shift_summary[run_mins] ),
    SUM( oee_shift_summary[planned_mins] ),
    0
)

-- OEE = A × P × Q (recalculated for slicer context)
OEE % =
VAR _A = DIVIDE( SUM(oee_shift_summary[run_mins]),  SUM(oee_shift_summary[planned_mins]), 0 )
VAR _P = DIVIDE( SUM(oee_shift_summary[units_total]) * AVERAGE(oee_shift_summary[actual_rate_avg]),
                 SUM(oee_shift_summary[run_mins]) * AVERAGE(oee_shift_summary[ideal_rate]), 0 )
VAR _Q = DIVIDE( SUM(oee_shift_summary[units_good]), SUM(oee_shift_summary[units_total]), 0 )
RETURN  _A * _P * _Q

-- Micro-stop count (from fact table via relationship)
Micro-Stop Count =
CALCULATE(
    COUNTROWS( fact_production_event ),
    fact_production_event[down_mins] < 5,
    dim_reason[is_planned] = FALSE()
)
